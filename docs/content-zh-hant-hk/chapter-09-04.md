## 🎯 核心目標
- 鞏固進階章節中的實踐心得。
- 獲取在構建或評估 LLM 驅動工具時可應用的知識。

:::callout-tldr
本節介紹了如何讓 LLM 真正發揮作用：為其提供在現實世界中行動的工具，將其與你的數據連接，使其基於知識而非猜測做出回答，並將其處理能力擴展到文本之外。最大的啟發在於：大多數「AI 失敗」實際上是餵給 LLM 的內容管理不善導致的 —— 有時負責餵入數據的工具才是真正的罪魁禍首。
:::

---

## 🧠 值得記住的 7 件事

**LLM 僅輸出文本 —— 由其他組件完成工作**
當 LLM「發送電子郵件」或「運行搜索」時，它實際上是在輸出一段格式化的指令，由周圍的系統攔截並執行。LLM 是大腦，你仍然需要「雙手」。

**MCP 是 AI 工具的 USB-C**
MCP (**模型上下文協議 (Model Context Protocol)**) 規範了 LLM 與外部工具及數據的連接方式。不再需要為每個工具進行自定義集成，一個協議即可連接所有工具 —— 就像 USB-C 取代了數十種專用充電器一樣。

**代理 (Agents) 在循環中運作，而非腳本**
優秀的代理 (Agents) 會循環執行：感知現狀 → 計劃行動 → 使用工具執行 → 觀察結果 → 重複。糟糕的代理則遵循僵化的腳本，一旦發生意外就會崩潰。

**RAG = 先搜索，後回答**
RAG 不是讓 LLM 憑記憶回答（這會產生幻覺），而是先在你的文檔中搜索相關信息，然後讓 LLM 根據搜索到的內容做出回答。搜索的質量與 LLM 本身同樣重要。

**背景資料工程 (Context engineering) 與模型選擇同樣重要 —— 且往往是問題所在**
大多數 AI 失敗都是背景資料 (Context) 方面的失敗 —— 糟糕的提示 (Prompt)、檢索到的文檔有誤、信息過多或過少。在責怪模型之前，請先檢查你餵給它的內容。

**「記憶」只是更好的背景資料 (Context)**
個人 AI 功能（如 ChatGPT Memory、Claude Projects）並未賦予 LLM 真正的記憶 —— 它們只是預加載了更多背景資料。模型每次仍然是從頭開始閱讀所有內容。

**理解媒介 ≠ 生成媒介**
LLM 可以準確描述照片中的內容，但生成寫實的照片是一項完全不同（且更難）的技能。不要假設具備其中一種能力就意味著具備另一種。

**LLM 使用的工具有其自身的局限性**
LLM 的可靠程度取決於它所依賴的工具。搜索引擎可能返回過時的結果，數據庫可能有舊紀錄，文件讀取器可能搞亂複雜的電子表格，轉錄工具可能聽錯詞句。當你評估一個由 LLM 驅動的系統時，不要只問「模型有多好？」，還要問「它依賴哪些工具，這些工具的盲點是什麼？」。鏈條的強度取決於最薄弱的一環 —— 而那一環往往不是 LLM 本身。

## 🚨 9 個危險信號

**「LLM 自己運行了搜索。」** —— 實際上是 LLM 輸出了一段文本指令，由它周圍的軟件攔截並執行。你使用的產品是模型與應用層的結合。模型決定做什麼，應用層決定是否做以及如何做。請不要混淆這兩者。

**代理 (Agents) 的失敗源於糟糕的計劃，而非工具。** —— 混亂的計劃會導致錯誤的行動。解決方法：從更清晰的提示 (Prompt) 開始，而不是更換更好的工具。

**在無人監督的情況下賦予自主代理 (Autonomous agents) 敏感訪問權限。** —— 始終從高度的人工監督開始。自主權應透過證明的可靠性來獲得，而非口頭承諾。

**舊問題在代理中依然存在。** —— 幻覺 (Hallucinations)、背景資料腐化和提示注入 (Prompt injection) 在代理循環的每一步都真實存在。給 LLM 封裝上工具並不能解決 LLM 本身的問題。

**RAG 減少但並未消除幻覺。** —— LLM 仍可能誤讀檢索到的文檔。良好的檢索質量是你需要監控的新指標。

**當你的數據可以裝進一個文件時卻去構建 RAG。** —— 如果你的知識庫小到可以直接粘貼到上下文窗口中，那就直接粘貼。RAG 會增加你可能並不需要的複雜性。

**你的數據格式是真正的瓶頸 —— 模型可能並未看到你上傳的所有內容。** —— 在你的文件與 LLM 的回應之間，平台可能運行了 OCR、剝離了圖像、扁平化了佈局，或僅從視頻中提取了字幕。表格丟失結構、多欄文本被打亂、圖表消失。LLM 會根據殘存的內容自信地回答 —— 且不會告訴你丟失了什麼。簡單的格式（如 Markdown、純文本）始終是首選。

**存儲錯誤信息的記憶系統比沒有更糟。** —— 過時或錯誤的事實會讓 LLM 自信地犯錯。有缺陷的記憶系統比從零開始更危險。

**工具出錯卻歸咎於 LLM。** —— 當 AI 系統給出錯誤答案時，LLM 的推理可能完全正確 —— 但它是基於錯誤、不完整或扭曲的工具輸出進行推理的。搜索引擎返回過時信息、數據庫有舊紀錄或文件讀取器搞亂了格式 —— 任何這些情況都會導致 LLM 輸出錯誤，即使模型本身完美無缺。在更換模型或重寫提示之前，請檢查工具究竟餵給了 LLM 什麼內容。

## 💡 專業建議

:::callout-tldr
**使用 LLM 來設計你的架構，而不僅僅是提示。** 既然你已經了解了工具、RAG 和背景資料工程 (Context engineering)，不妨讓 LLM 幫助設計架構本身。例如：「我希望 LLM 回答產品文檔中的問題。我應該使用 RAG、直接粘貼所有內容，還是進行微調？請根據我的限制條件幫我做決定。」
:::

LLM 可以幫助你分析權衡、設計背景資料注入策略，甚至起草工具配置 —— 而不僅僅是編寫提示。將其視為設計夥伴，而不僅僅是文本生成器。

## 📐 快速參考

**RAG 對比 微調 (Fine-Tuning)** —— 從提示開始。如果不足夠，再加入 RAG（提供你的文檔）。只有當知識確實是專有的且任何公開模型都無法得知時，才進行微調。 ([瞭解更多](chapter-07-07.html))

**RAG 的替代方案** —— 不要默認使用 RAG。關鍵字搜索快速且便宜。對於小型數據集，直接粘貼所有內容即可。子代理 (Subagents) 可提供深度理解。選擇最簡單有效的方法。 ([瞭解更多](chapter-07-04.html))

**代理 (Agent) 配置的 4 個槓桿** —— **工具**（越少越安全）、**領域**（限定範圍）、**訪問權限**（默認唯讀）、**角色**（賦予明確的職位名稱）。未經配置的代理就是失控的代理。 ([瞭解更多](chapter-06-06.html))

**金髮姑娘背景資料 (Goldilocks Context)** —— 背景資料太少 → LLM 會猜測。太多 → 質量會因噪音而下降。目標是 2-3 個高度相關的章節，而非 30 頁關係疏遠的內容。 ([瞭解更多](chapter-07-05.html))

**AI 的三項權衡** —— 更實用但準確率降低（錯誤的背景資料）。更個性化但隱私減少（共享更多數據）。更強大但風險更高（行動 vs. 唯讀）。每一分收穫都有其代價。 ([瞭解更多](chapter-08-04.html))
