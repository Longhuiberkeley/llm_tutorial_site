## 🎯 核心目標
- 描繪 AI 輔助開發的頻譜 —— 從純粹的「氛圍編程」到結構化的、人類主導的工作。
- 定義氛圍編程 (Vibe coding)，並解釋為什麼它在 LLM 輔助工作中會自然出現。
- 展示沒有結構的速度如何在任何類型的產出（不僅僅是代碼）中產生偏移 (Drift)。
- 介紹 SPEC Kit 和 BMAD，作為將工作錨定在真實需求上的實用結構化框架。

:::callout-tldr
「氛圍編程 (Vibe coding)」感覺很對。速度令人陶醉。但在沒有結構的情況下快速生成，會讓看起來令人印象深刻的項目變得無法維護。結構化框架並非官僚主義 —— 它們是讓速度可持續的腳手架。
:::

## 📊 AI 輔助開發的頻譜

並非所有的 LLM 輔助工作都是一樣的。這裡有一個頻譜。

**一端 —— 純粹的氛圍編程 (Vibe coding)。** 普及這一術語的研究員 Andrej Karpathy 將其描述為：完全聽從「感覺 (Vibes)」，擁抱指數級增長，甚至忘記代碼本身的存在。在這種模式下，程式設計師負責引導、測試和提供反饋，而不是手動編寫代碼。LLM 負責編寫；人類負責導航。Karpathy 在早前的觀察中捕捉到了這種精神：最新的熱門編程語言是英語 —— 即 LLM 已強大到足以讓自然語言指令取代明確的代碼。

**另一端 —— AI 輔助的匠人模式 (AI-assisted craftsmanship)。** LLM 是你諮詢的工具，而不是你委託的系統。代碼補全、針對性的建議和偶爾的代碼生成會輔助你的工作 —— 但你理解每個決策，大部分內容是由你編寫的，LLM 從未對任何實質性組件擁有「作者身份」。質量反映的是你的判斷力；LLM 只是放大了它。

**中間 —— 結構化 AI 開發 (Structured AI development)。** 你大量使用 LLM —— 有時甚至是針對整個模組或章節 —— 但前提是先寫好需求、每個 Prompt 都有清晰的上下文，並在接受前進行真實的審核。你的角色更像是經理或架構師，而不是一線編寫者：指導工作、檢查產出、做出決策並修正航向。這正是 SPEC Kit 和 BMAD 等框架的用武之地。

大多數實際工作都落在中間。問題不在於是否使用 LLM —— 而在於你在使用過程中引入了多少結構和審核。

## 🎸 什麼是氛圍編程 (Vibe coding)？

**氛圍編程 (Vibe coding)** 主要是指通過向 LLM 發送 Prompt、快速迭代產出，並接受看起來「行得通」的結果來進行構建 —— 而沒有深入理解已構建的內容，也沒有根據設定的需求進行驗證。

這個名字源於軟件開發，但這種模式適用於任何 LLM 輔助的工作。例如：在沒有清晰論證結構的情況下，通過一段接一段的生成來撰寫 40 頁的行業報告。通過隨意添加想到的功能來構建應用程序。在沒有定義研究問題的情況下，通過 Prompt 生成內容來製作研究文件。

在每種情況下，LLM 都有幫助。產出看起來不錯。問題則在後期顯現。

核心問題在於**偏移 (Drift)**：沒有清晰的規範，你構建的是當時覺得有趣的東西，而不是真正需要的東西。LLM 並不知道你的目標 —— 它針對你的 Prompt 優化出聽起來很合理的回答。如果 Prompt 沒有錨定在需求上，產出的內容也不會錨定在需求上。

當木匠拿到一把射釘槍時，第一反應往往不是更仔細地規劃 —— 而是飛快地在所有地方都釘上釘子。LLM 對項目也有同樣的影響。幾秒鐘內就能生成內容的能力創造了一種令人陶醉的速度。團隊行動迅速，演示看起來令人印象深刻，然後有人問：「這真的解決了我們最初想要解決的問題嗎？」

:::callout-dyk
軟件工程師 Ward Cunningham 在 1992 年創造了「技術債務 (Technical debt)」一詞，用來描述為了眼前的快速簡便而放棄長遠更好方法的代價。氛圍編程會加速技術債務的累積 —— 但這個概念同樣適用於報告、文件和研究。例如：用不同措辭重覆相同分析的幻影段落。到第 8 節時與自身矛盾的論證結構。當你嘗試編輯、發布或根據產出採取行動時，你就是在支付這筆債務的利息。
:::

## 🏎️ 速度陷阱：質量的退化

氛圍編程的危險不會立即顯現，它有一定的延遲性。

**第 2 週：** 一切都很好。功能或章節快速出現。項目感覺充滿活力且高效。

**第 6 週：** 添加新內容比預期花費更長的時間。修改一個章節會破壞另一個章節的邏輯。一個新功能產生了三個新 Bug。項目開始變得「粘手」。

**第 12 週：** 本應是一天就能完成的添加工作，現在需要三天的「考古」。你花在理解已構建內容上的時間，比構建新東西的時間還要多。

這種模式在各類項目中都一致存在。一份在一個月內通過氛圍編程撰寫的 50 頁行業報告，在草稿階段看起來很驚人。但到了需要編輯時 —— 當一個核心論點需要改變，或者需要納入新的發現時 —— 缺乏底層結構會讓每次編輯都充滿風險。修改一個章節會揭露出其他兩個章節的不一致之處。

解決方法不是停止使用 LLM，而是在**生成之前投入時間建立結構**。設計大綱、定義論點、明確需求 —— 然後利用 LLM 在該結構內產出內容。

## 📋 解毒劑：結構化框架

為了應對 LLM 輔助開發中的氛圍編程，出現了幾種框架。它們的核心方法一致：在生成任何內容之前，先清晰地規範工作。

### SPEC Kit (規格套件)

**SPEC Kit** 是一個開源工具包，支持規格驅動開發 (Spec-Driven Development, SDD)。它不使用模糊的 Prompt，而是使用結構化的 Markdown 文件（如 `spec.md`, `plan.md`）為 LLM 代理提供清晰的上下文、需求和約束。

SPEC Kit 工作流定義了一個清晰的生命週期：設定**憲章 (Constitution)**（項目的規則和原則）、編寫**規格 (Specification)**、制定**技術方案**、生成**任務**，最後才是**實施**。每個階段都是明確的移交 —— 在沒有前一階段產出的情況下，你不能跳到下一階段。

其核心是一個強迫具體化的需求模板。對於你想構建或撰寫的每個功能：

---

**作為一名** [用戶角色]
**我需要** [能力]
**以便** [達成結果]

**輸入 (INPUT)：** 進入該功能或章節的是什麼？
**輸出 (OUTPUT)：** 出來的是什麼？具體應產出什麼？
**約束 (CONSTRAINTS)：** 適用哪些限制和規則？絕對不能做什麼？
**成功標準 (SUCCESS CRITERIA)：** 我們如何知道它運作正常或已完成？

---

**示例：** 「作為一名客戶支援專員，我需要 LLM 起草回覆郵件，以便我能更快回覆客戶。輸入：客戶郵件 + 公司政策文件。輸出：一份可供編輯和發送的回覆草稿。約束：在提出主張時必須引用具體政策；超過 100 美元的退款需經經理批准。成功標準：80% 的草稿只需微調或無需修改即可使用。」

注意填寫這個模板需要什麼：你必須了解你的用戶、輸入、預期輸出、規則和成功指標。這正是你在生成任何內容之前需要的資訊 —— 也是氛圍編程所跳過的資訊。

:::callout-error
一個常見錯誤：撰寫需求時描述的是技術（「使用此 LLM 分析郵件」）而不是業務需求（「將平均回覆時間從 4 小時縮短至 30 分鐘」）。需求應該描述問題。解決方案可以變 —— 但問題定義了成功。
:::

### BMAD (構建更多建築師之夢)

**BMAD** (Build More Architect Dreams) 是一個免費的開源敏捷框架，專為 LLM 輔助開發而設計。BMAD 不讓一個代理處理所有事情，而是將工作組織成一個多代理團隊：產品經理、架構師、開發人員和審核員 —— 每個角色都有其界限。給予每個代理清晰、有限的職責，能減少幻覺並提高整個開發生命週期的精確度。

BMAD 的思維模式是每個功能在構建前都必須回答的「問題鏈」：

1. **為什麼要構建這個？** —— 它解決了什麼問題？
2. **成功是什麼樣子的？** —— 一個可衡量的結果，而非一種感覺。
3. **誰會受益？** —— 哪個用戶或團隊，以及如何受益？
4. **不構建這個的代價是什麼？** —— 它是緊急的還是「有了更好」的？

SPEC Kit 和 BMAD 可以協同工作：SPEC Kit 捕捉「要構建什麼」的需求，而 BMAD 確保每一項都值得去規範。

### 其他框架

GSD、Superpowers 以及日益增長的類似框架生態系統都採取了同樣的根源方法：生成前先定義，提速前先結構化。具體工具並不重要，重要的是使用工具的紀律。任何能強迫你在打開 Prompt 窗口前回答「我正在構建什麼、為誰構建、我如何知道它完成了」的框架，產出的結果都會優於完全沒有框架。

## 📝 核心概念

- **頻譜** —— LLM 輔助工作從純粹的氛圍編程（LLM 編寫，人類引導）到結構化 AI 開發（LLM 在定義的規格內生成，人類導航）。大多數實際工作處於中間。
- **氛圍編程 (Vibe coding)** = 通過快速 Prompt 並接受看起來可行的產出來進行構建，而沒有錨定在需求上。適用於代碼、報告、文件 —— 任何 LLM 產出。
- **偏移 (Drift)** —— 沒有規格，你構建的是當時感興趣的東西而非真正需要的。速度掩蓋了問題，直到修復成本變得昂貴。
- **SPEC Kit** = 規格驅動開發的開源工具包。使用結構化模板（用戶/能力/結果/輸入/輸出/約束/成功標準）將每個功能錨定在真實需求上。
- **BMAD** = 多代理開發框架。將 LLM 工作組織成專業角色（PM、架構師、開發、審核），以減少幻覺並提高質量。
- **任何一致的框架都優於沒有** —— 關鍵在於在生成前回答「什麼、為誰、如何知道已完成」的紀律。
